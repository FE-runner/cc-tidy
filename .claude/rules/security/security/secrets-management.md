# 密钥管理

## 核心原则

- 永远不要将密钥提交到版本控制 — 无例外
- 关键密钥最长每 90 天轮换一次
- 最小权限 — 仅授予必要的访问权限
- 静态和传输中加密
- 在应用启动时验证必需的密钥 — 缺失时快速失败

## 环境文件

| 文件 | 提交？ | 用途 |
|------|--------|------|
| `.env` | 否 | 本地环境变量（可能包含密钥） |
| `.env.example` | 是 | 带占位符值的模板（提交） |
| `.env.local` | 否 | 带密钥的本地覆盖 |
| `.env.development` | 是 | 开发环境（无密钥） |
| `.env.production` | 是 | 生产模板（无密钥） |
| `.env.*.local` | 否 | 环境密钥 |

## .gitignore 强制要求

- 始终忽略：`.env.local`、`.env.*.local`、`*.pem`、`*.key`、`**/secrets/`、`credentials.json`、`service-account.json`

## 密钥存储决策

| 类型 | 存储 |
|------|------|
| API 密钥、数据库凭证、OAuth 密钥 | Vault / 云密钥管理器 |
| JWT 签名密钥 | Vault / 环境变量（永远不在代码中） |
| 加密密钥 | KMS / HSM |
| TLS 证书 | 证书管理器 / 云提供商 |
| CI/CD 密钥 | 平台密钥存储（GitHub Actions Secrets、GitLab CI Variables） |

## 密钥轮换流程

1. 生成新密钥
2. 部署同时接受新旧密钥（双读）
3. 更新所有消费者使用新密钥
4. 从配置中移除旧密钥
5. 在提供商中撤销旧密钥

## 反模式

- 禁止将 `.env` 连同真实密钥一起提交 — 使用 `.env.example` 作为模板
- 禁止通过 Slack/邮件共享密钥 — 使用密钥管理器共享
- 禁止跨环境使用相同的密钥 — 开发环境被攻破不应影响生产环境
- 禁止在 Dockerfiles 或 CI 配置中硬编码密钥 — 使用挂载的密钥
- 禁止意外记录密钥 — 在序列化器中编辑敏感字段
- 禁止在泄露后跳过轮换 — 立即轮换

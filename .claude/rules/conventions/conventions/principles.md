# 软件工程原则

## YAGNI — 不要为假设的未来而构建

- 只实现当前需要的，而非将来可能需要的
- 禁止创建只有一个实现的抽象基类
- 禁止为只有一个值的东西添加配置
- 在需求出现时重构，而非提前

## KISS — 最简单的可行方案

- 优先选择普通函数而非单例类
- 优先选择显式代码而非巧妙的抽象
- 如果难以解释，那就太复杂了 — 简化它

## SoC — 分离验证、业务逻辑、数据访问

- 验证属于边界层（controller/handler）
- 业务逻辑属于服务层
- 数据访问属于仓库层
- 禁止将 HTTP 关注点混入服务层

## DRY — 但重复好过错误的抽象

- 仅在上下文真正相同时提取共享代码
- 两个有不同领域上下文的相似函数不算重复
- 三次法则：容忍重复直到第三次出现，然后再抽象

## 决策矩阵

| 原则 | 应该 | 不应该 |
|------|------|--------|
| YAGNI | 构建当前需要的 | 添加"以防万一"的功能 |
| KISS | 编写简单、可读的代码 | 过度设计使用模式 |
| SoC | 保持层之间独立 | 混合 HTTP、业务和数据逻辑 |
| DRY | 提取真正共享的逻辑 | 将不相关的代码强行塞入一个抽象 |

# 可观测性

## 结构化日志（JSON）

- 所有日志必须是结构化 JSON — 生产环境中禁止纯文本
- 包含：`timestamp`、`level`、`message`、`service`、`traceId`、上下文字段
- 使用 OpenTelemetry 作为分布式追踪和遥测收集的标准

### 日志级别

| 级别 | 何时使用 |
|------|----------|
| `error` | 需要关注的故障（告警） |
| `warn` | 可恢复的问题、弃用 |
| `info` | 业务事件、状态变更、请求生命周期 |
| `debug` | 开发细节 — 在生产环境中禁用 |

### 应该记录什么

- 请求接收/完成及耗时
- 业务事件（用户创建、订单下单、支付处理）
- 外部服务调用及耗时和状态
- 带堆栈跟踪和上下文的错误
- 认证事件（登录、登出、失败尝试）

### 永远不要记录

- 密码、token、API 密钥、密钥
- 信用卡号、PII、PHI
- 生产环境中的完整请求/响应体
- 健康检查流量（污染日志）

## 关联 ID

- 在整个请求链中生成或传播 `x-trace-id` 头
- 在每个日志条目中包含 `traceId` — 分布式调试的必要条件
- 将 trace ID 传递给所有下游服务调用

## 健康检查

- `/health/live` — 进程是否在运行？（始终 200）
- `/health/ready` — 能否服务流量？（检查数据库、缓存、依赖）

## 指标（RED 方法）

| 指标 | 追踪内容 |
|------|----------|
| **速率** | 每个端点的请求/秒 |
| **错误** | 错误率（5xx / 总数） |
| **持续时间** | 延迟 p50、p95、p99 |

## 反模式

- 禁止记录敏感数据 — 即使通过对象展开意外记录
- 禁止跳过关联 ID — 使跨服务调试变得不可能
- 禁止不记录日志就吞掉错误 — `catch {}` 是一个 bug
- 禁止向用户暴露堆栈跟踪 — 内部记录日志，返回通用消息
- 禁止不带上下文记录日志 — "发生错误"没有用处；包含实体 ID、操作、用户
